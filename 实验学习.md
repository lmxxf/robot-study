# 实验学习

---

## 实验框架：这堆软件各自干什么

```
┌─────────────────────────────────────┐
│  你（敲命令的人）                      │
├─────────────────────────────────────┤
│  LeRobot（框架/胶水）                  │  ← 串起一切的"导演"
├──────────┬──────────────────────────┤
│  MuJoCo  │  AI 模型（大脑）           │
│ （物理引擎）│  Diffusion / ACT / ...   │  ← 虚拟世界 + 决策者
├──────────┴──────────────────────────┤
│  PyTorch（深度学习引擎）               │  ← 模型跑在这上面
└─────────────────────────────────────┘
```

| 软件 | 角色 | 类比 |
|------|------|------|
| **MuJoCo** | 物理仿真引擎 | 游戏引擎——造了一个有物理规则的虚拟桌面，积木有质量、有摩擦力、会被推动 |
| **AI 模型**（Diffusion Policy / ACT / SmolVLA） | 决策大脑 | 考生——看画面，决定怎么动 |
| **LeRobot** | 框架 | 监考老师——把考场（MuJoCo）和考生（模型）对接起来，喊开始、收卷子、算分 |
| **PyTorch** | 计算引擎 | 考生大脑的底层硬件——模型的矩阵运算跑在这上面，用 GPU 加速 |
| **gym-pusht / gym-aloha** | 环境包装 | 考卷模板——定义具体题目的规则：初始位置怎么随机、成功怎么判定、一局多少步 |

### 跑 eval 时到底发生了什么

```
1. LeRobot 读你的命令，知道要考 Push-T，用 Diffusion Policy

2. LeRobot 叫 gym-pusht 出题 → gym-pusht 叫 MuJoCo 造虚拟桌面
   → 随机摆好积木和手指位置

3. 循环 300 步：
   a. MuJoCo 渲染当前画面 → 给模型看
   b. Diffusion Policy（跑在 PyTorch 上）看画面 → 输出动作
   c. LeRobot 把动作交给 MuJoCo → MuJoCo 模拟物理效果 → 手指移动、积木被推
   d. gym-pusht 判断积木到位了没 → 算 reward

4. 300 步跑完，LeRobot 判定成功/失败，录视频，下一局

5. 10 局跑完，LeRobot 汇总成功率：60%
```

**一句话：MuJoCo 是考场，模型是考生，LeRobot 是监考老师。** 你敲的那行命令就是跟监考老师说："用这个考生，考这道题，考 10 局。"

## 实验一：Push-T（Diffusion Policy 推积木）

### 实验概况

- **任务**：一个圆形"手指"把 T 形积木推到目标位置
- **模型**：Diffusion Policy（扩散策略），别人用模仿学习训好的
- **结果**：10 局考试，成功率 60%，GPU 下 71 秒跑完

---

### 数据长什么样

人类用鼠标控制圆球推积木，每一帧录下来：

```
时间步 t=0:  观测=[摄像头画面, 手指位置(x,y)]  动作=[手指移动(dx,dy)]
时间步 t=1:  观测=[摄像头画面, 手指位置(x,y)]  动作=[手指移动(dx,dy)]
...
时间步 t=299: 观测=[摄像头画面, 手指位置(x,y)]  动作=[手指移动(dx,dy)]
```

一局 300 步，录了几百局。本质就是一堆 **(看到什么, 做了什么)** 的配对。

---

### 模型学什么

模型要学的函数很简单：

```
f(当前画面, 手指位置) → 接下来手指怎么移动
```

输入是"看到的"，输出是"该做的"。和训 LLM 本质一样——LLM 学的是 `f(上文) → 下一个token`，这里学的是 `f(观测) → 下一段动作`。

---

### Diffusion Policy 怎么生成动作

不是直接输出一个动作，而是用**扩散模型**（和画图的 Stable Diffusion 同源）：

```
纯噪声 [随机数, 随机数, ...]
  ↓ 去噪第1步（条件：当前画面+手指位置）
  ↓ 去噪第2步
  ↓ ...
  ↓ 去噪第100步
干净的动作序列 [dx₁,dy₁, dx₂,dy₂, ... dx₁₆,dy₁₆]
```

**一次生成未来 16 步的动作**（action chunk），不是一步一步蹦。这就是 config 里 `horizon: 16` 的意思。但只执行前 8 步（`n_action_steps: 8`），然后重新看一眼画面，再生成下一段。

**为什么用扩散而不是直接回归？** 因为推积木的"正确路径"不止一条——可以从左边推，也可以从右边推。直接回归会把两条路径平均成一条（从中间推，废了）。扩散模型每次采样会随机选一条可行路径，不会平均。这叫**多模态分布问题**（multimodal distribution），是机器人控制里的核心难题。

---

### 模型结构

```
摄像头画面 (384×384)
    ↓ ResNet18 提取特征 → 32个关键点坐标
    ↓
手指位置 (x,y)
    ↓
拼在一起 → 条件向量
    ↓
1D U-Net（去噪网络）
    ↓
输出：16步动作序列 (16×2 = 32个数字)
```

整个模型约 **2.6 亿参数**（config 里 `down_dims: 512,1024,2048` 的 U-Net）。对比：GPT-2 是 1.5 亿。推个积木用的参数量和 GPT-2 一个量级——模型不是瓶颈，数据才是。

---

### 关键参数解读

| 参数 | 值 | 含义 |
|------|-----|------|
| `horizon` | 16 | 一次生成未来 16 步动作 |
| `n_action_steps` | 8 | 只执行前 8 步，然后重新观测 |
| `n_obs_steps` | 2 | 看最近 2 帧画面做决策（不是只看当前帧） |
| `num_train_timesteps` | 100 | 扩散去噪 100 步 |
| `vision_backbone` | ResNet18 | 用 ResNet18 提取画面特征 |
| `down_dims` | (512, 1024, 2048) | U-Net 各层通道数，决定模型大小 |

---

### 和 LLM 的类比

| | LLM | Diffusion Policy |
|---|---|---|
| 输入 | 上文 token 序列 | 摄像头画面 + 状态 |
| 输出 | 下一个 token | 未来 16 步动作 |
| 训练数据 | 人类写的文本 | 人类示范的轨迹 |
| 生成方式 | 自回归（一个个蹦） | 扩散（从噪声去噪） |
| 核心问题 | 一句话有多种合理续写 | 一个场景有多种合理动作 |

**本质都是"从人类行为数据中学习条件概率分布"。** LLM 学 P(下一个词|上文)，这个模型学 P(动作序列|观测)。

---

### 为什么只有 60% 成功率

1. **训练数据有限**：人类示教只录了几百局，覆盖不了所有初始位置
2. **没有在线纠错**：模型执行 8 步后才重新看画面，中间推歪了没法及时修正
3. **扩散采样的随机性**：每次从噪声开始，偶尔采到不好的路径
4. **这已经是不错的成绩**：60% 是预训练 demo 的典型水平，自己训练调参可以更高
